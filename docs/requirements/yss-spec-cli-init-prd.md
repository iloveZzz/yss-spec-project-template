# 产品需求文档：yss-spec 模板初始化 CLI

## 问题陈述

当前 `yss-spec-project-template` 作为研发管理模板源仓库，已经具备 PRD、架构、OpenAPI、Issue 和验证流程等可复用资产，但初始化新项目时仍主要依赖手工复制与人工修改。对于项目初始化负责人和内部开发者 / Tech Lead 来说，这种方式存在三类核心问题：

1. 初始化步骤不一致，不同人会复制不同内容、删除不同目录、保留不同示例资产，导致模板实例仓库质量不稳定。
2. 项目级元信息替换容易遗漏，例如项目名称、业务领域、团队规模、Issue 平台偏好等占位信息可能残留在实例仓库中。
3. 模板源仓库包含维护性目录、技能资产和本地工具配置，缺少明确的复制 / 渲染 / 排除规则时，实例仓库容易混入不该暴露给最终项目的内容。

结果是：团队虽然有模板，却没有一个低门槛、可重复、可验证的初始化入口，导致新项目落地效率和模板一致性都受影响。

## 解决方案

提供一个基于 npm 分发的“模板初始化 CLI”，把“复制模板 + 收集项目参数 + 渲染必要文件 + 执行安全检查 + 可选 Git 初始化”收敛为统一命令入口。

用户通过 `npm create yss-spec@latest` 或 `npx create-yss-spec@latest` 启动 CLI，按提示输入项目名、业务领域、目标目录等最小必要信息。CLI 根据模板清单将模板源仓库内容划分为三类：

- `render`：需要替换项目级变量的文件
- `copy`：原样复制的模板资产
- `exclude`：不应进入模板实例仓库的维护性文件

CLI 在落盘前提供 `--dry-run` 预览能力，在目标目录非空时默认 阻断项闭环，并支持显式 `--force` 覆盖。初始化完成后，可选执行 `git init`，让用户直接从一个干净、规范、可继续进入生命周期流程的模板实例仓库开始工作。

## 用户故事

1. 作为项目初始化负责人，我希望通过一个统一命令创建模板实例仓库，以便我不用再手工复制和筛选模板文件。
2. 作为项目初始化负责人，我希望 CLI 能交互式询问项目名称，以便生成后的仓库元信息准确可用。
3. 作为项目初始化负责人，我希望 CLI 能交互式询问业务领域，以便模板中的项目说明和上下文信息更贴近真实业务。
4. 作为项目初始化负责人，我希望 CLI 能支持目标目录参数，以便我可以把仓库初始化到指定位置。
5. 作为项目初始化负责人，我希望 CLI 在目标目录非空时默认停止，以便避免误覆盖已有文件。
6. 作为项目初始化负责人，我希望在明确知道风险时使用 `--force` 覆盖，以便在受控场景下重复初始化。
7. 作为项目初始化负责人，我希望在真正写入文件前先运行 `--dry-run`，以便确认将要复制、渲染和排除的内容。
8. 作为项目初始化负责人，我希望 CLI 能自动排除模板维护目录，以便生成结果只包含实例项目真正需要的资产。
9. 作为项目初始化负责人，我希望 CLI 只渲染必要文件，其余模板资产原样保留，以便既能定制项目信息，又不破坏模板结构。
10. 作为项目初始化负责人，我希望 CLI 能在完成后可选执行 `git init`，以便我立即进入版本管理流程。
11. 作为内部开发者，我希望命令形态符合 npm 生态习惯，以便我无需额外学习定制化工具入口。
12. 作为内部开发者，我希望 CLI 提供清晰的成功输出和下一步提示，以便我知道仓库已经初始化完成以及接下来该做什么。
13. 作为内部开发者，我希望生成后的 `AGENTS.md`、README 等项目元信息文件不再保留模板占位符，以便减少后续人工清理工作。
14. 作为内部开发者，我希望 CLI 能收集团队规模等可选元信息，以便文档初始状态更完整。
15. 作为内部开发者，我希望 CLI 能让我选择默认 issue tracker 偏好，以便后续协作文档有合理初值。
16. 作为内部开发者，我希望 CLI 不强制安装依赖，以便工具在非 runtime 仓库场景下保持轻量。
17. 作为内部开发者，我希望 CLI 不自动创建远端仓库或 CI 配置，以便把组织级权限操作留在人工确认后进行。
18. 作为产品 / 架构协作者，我希望模板实例仓库从一开始就包含规范化的 PRD、架构和 Issue 资产，以便更快进入正式流程。
19. 作为模板维护者，我希望模板初始化逻辑和模板源仓库同版本演进，以便避免 CLI 版本和模板内容漂移。
20. 作为模板维护者，我希望复制规则由清单驱动，而不是散落在代码中，以便后续维护更可控。
21. 作为模板维护者，我希望能通过行为测试验证“同样输入生成同样输出”，以便保证模板初始化的一致性。
22. 作为模板维护者，我希望能验证维护性目录不会被误复制，以便降低实例仓库被污染的风险。
23. 作为模板维护者，我希望工具优先服务单模板初始化场景，以便先打稳核心路径，而不是过早做多模板平台。
24. 作为模板维护者，我希望 npm 包可以先用于内部私有分发，以便在发布范围可控的情况下快速迭代。
25. 作为未来的使用者，我希望 CLI 在失败时能告诉我是哪一步失败、哪些文件未写入，以便我能快速重试或定位问题。

## 功能需求

| ID | 需求 | 优先级 | 备注 |
|---|---|---|---|
| FR-001 | 系统必须提供 `npm create yss-spec@latest` 对应的 CLI 初始化入口。 | P0 | 可兼容 `npx create-yss-spec@latest` |
| FR-002 | 系统必须支持交互式收集 `projectName`、`businessDomain`、`targetDir` 等必要参数。 | P0 | 可选参数见 FR-003 |

| FR-005 | 系统必须对 `render` 文件执行变量替换，对 `copy` 文件原样复制，对 `exclude` 文件不落盘。 | P0 | 变量来源于用户输入与默认值 |
| FR-006 | 系统必须在目标目录非空且未显式授权覆盖时拒绝执行写入。 | P0 | 默认安全 |
| FR-007 | 系统必须支持 `--force` 覆盖策略。 | P1 | 仅在显式传参时启用 |
| FR-008 | 系统必须支持 `--dry-run`，展示将要执行的复制、渲染和排除结果。 | P0 | 不应产生真实写入 |
| FR-009 | 系统应支持初始化完成后可选执行 `git init`。 | P0 | 不负责远端仓库创建 |
| FR-010 | 系统必须在初始化完成后输出成功摘要和下一步建议。 | P1 | 至少包含进入仓库、检查文档、开始 Git 的提示 |
| FR-011 | 系统必须确保生成结果中关键元信息文件不残留模板占位信息。 | P0 | 关键文件包括 `AGENTS.md`、README 等 |
| FR-012 | 系统不应在 MVP 中自动安装依赖、创建远端仓库或初始化 CI。 | P0 | 明确保持非目标范围 |
| FR-013 | 系统应支持以当前仓库同版本演进的方式维护模板与 CLI。 | P1 | 影响实现位置与发布方式 |

| 类别 | 需求 | 目标 |


## 测试决策

- 主要测试 seam：CLI 端到端行为测试。通过执行命令并检查输出目录结构、渲染结果和命令返回值来验证外部行为，而不是验证内部实现细节。
- 次级测试 seam：模板清单解析与渲染规则的纯函数测试，仅在端到端测试无法高效覆盖边界条件时补充。
- 代码库中的相似测试：当前仓库未发现现成 npm CLI 或等价测试先例，需为该工具建立新的行为测试基线。
- 好测试的标准：
  - 只验证用户可观察行为，例如生成了哪些文件、哪些文件被排除、关键文本是否被正确替换、错误消息是否符合预期。
  - 避免断言内部函数调用次数、内部目录遍历顺序或具体实现细节。
  - 优先在最高 seam 验证完整初始化路径，再用少量下层测试覆盖难以稳定复现的边界条件。
- 必需测试：
  - [x] 单元 / 领域行为
  - [ ] API / 契约
  - [ ] 前端组件
  - [x] E2E 关键路径
- 建议覆盖的关键场景：
  - 成功初始化空目录
  - `--dry-run` 不落盘
  - 非空目录默认拒绝
  - `--force` 覆盖
  - `git init` 可选执行
  - `exclude` 目录未出现在输出中
  - `render` 文件中的关键占位符被正确替换

## 非目标范围

- 生成前端 / 后端运行时代码工程
- 更新已有模板实例仓库到最新模板版本
- 自动创建远端 Git 仓库、CI、Issue board 或其他外部平台资源
- 自动安装依赖或执行项目特定 bootstrap 流程
- 支持多模板市场或多产品线模板切换
- 把生命周期后续阶段资产（例如 PRD 正文、OpenAPI Freeze、垂直切片 Issue）在初始化时自动生成完成

## 风险

- 模板处理边界定义不清时，实例仓库可能误带维护性目录，或遗漏应复制的核心流程资产。
- 如果 CLI 与模板内容分仓维护，后续会出现版本漂移和验证成本上升。
- 包名、私有 npm registry 策略和版本发布节奏若未及时明确，可能影响首版可用性和推广方式。
- 若过早把“初始化工具”扩展成“多模板平台”，会显著抬高 MVP 复杂度并稀释核心价值。

## 实现决策摘要

- 该工具定位为“模板初始化 CLI”，而不是业务运行时代码脚手架。
- 实现位置优先建议在当前模板源仓库内，与模板同版本演进。
- 推荐包名为 `create-yss-spec`，并以 `npm create yss-spec@latest` 作为首选入口。
- 推荐通过模板清单驱动 `render` / `copy` / `exclude` 规则，而不是在代码中散落硬编码判断。
- 推荐默认面向内部受控分发场景，先支持私有 npm registry 或受控发包流程。
