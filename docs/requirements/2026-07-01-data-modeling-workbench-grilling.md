## 2. 用户与场景

| 用户 | 主要任务 | 当前痛点 | MVP 要解决的结果 |
|---|---|---|---|
| 数据架构师 | 规划主题域、定义业务对象和逻辑模型、制定规则、评审模型变更。 | 模型散落在文档和表结构中，开发落地后容易和逻辑口径不一致；评审时难看清变化。 | 可以在工作台内定义模型、查看映射覆盖、检查规范问题、基于 diff 完成评审。 |
| 数仓开发工程师 | 基于逻辑模型设计物理表、字段映射、DDL 草案和交接资产。 | 需要反复从文档理解业务含义，字段命名和类型靠人工约定，DDL 与模型文档容易不一致。 | 可以按逻辑模型创建物理表映射，提交规范检查和评审，发布后获得可交接资产。 |
| 数仓负责人 / 评审人 | 确认模型变更是否符合分层、命名、敏感字段和交付规范。 | 缺少统一的发布记录和可追溯版本。 | 可以查看评审记录、发布快照和变更说明。 |

## 3. MVP 范围

### 3.1 P0 必做

| 模块 | MVP 能力 | 说明 |
|---|---|---|
| 模型项目与主题域 | 创建模型项目，维护项目负责人、项目边界、主题域边界和业务对象列表。 | 支撑架构师从业务模型项目开始组织模型，而不是直接建表。 |
| 逻辑模型 | 定义逻辑实体、逻辑字段、字段语义、数据类型、是否必填、敏感标记、待分级状态和实体关系说明。 | 关系表达先支持轻量关系说明，不做复杂 ER 画布强依赖；正式敏感分级体系后置。 |
| 物理模型映射 | 定义物理表、字段、分层、主键、分区、逻辑字段映射和映射覆盖率。 | P0 先支持手工创建和模板导入字段清单，不要求直连数据库自动采集。 |
| 规范检查 | 检查命名、分层、字段类型、注释、必填元数据、敏感标记、待分级状态和映射完整性。 | 规则先做小而明确的可配置集合，支持阻断和警告两类结果。 |
| 模型评审 | 提交评审、评论、驳回、通过、查看变更 diff 和检查结果。 | 评审流先支持单模型项目内的轻量流程。 |
| 版本发布 | 生成发布快照、版本号、发布说明、模型文档和 DDL 草案。 | DDL 只作为草案导出，不自动执行数据库迁移。 |
| 交付资产 | 导出模型说明、字段清单、映射表、规范检查结果、DDL 草案。 | P0 同时支持 SQL 文本、Markdown 文档和 Excel 字段清单三种交付形式。 |
| 既有表导入 | 导入已有表结构作为物理模型草稿。 | P0 优先支持从 Excel 字段清单导入，后续再扩展数据库元数据直连采集。 |

### 3.2 试点模型项目建议

工作台会承载多个业务数据模型项目，“绩效风控指标模型复用”只是首个 P0 试点模型项目，不能把“指标平台全部模型”一次性纳入 MVP。根据 `指标平台表模型.xls` 的目录和字段映射结构，建议采用四级边界：

| 层级 | 建议名称 | 范围 | 建议 |
|---|---|---|---|
| 模型项目 | 绩效风控指标模型复用 | 围绕绩效风控 3.0 既有表、指标平台表模型和数据中台目标表之间的模型复用、字段映射和交付资产生成。 | 作为 P0 试点模型项目。 |
| 主题域 | 绩效风控 | 限定在绩效、收益、基准、资产类、证券收益等模型资产，不扩展到全量指标平台。 | 作为试点项目内的业务边界。 |
| 首批子域 | 产品绩效收益模型 | 每日产品净值收益、产品及基准收益率、产品资产类持仓收益、产品证券持仓收益，以及必要的产品、日期、证券主数据。 | 作为第一条端到端验证主线，范围适中，能覆盖逻辑模型、物理映射、规范检查、评审发布和导出。 |
| 后续子域 | 交易统计与市场基础指标模型 | 资产类交易统计、证券交易统计、指数收益率、基准收益率、证券行情、债券/股票市场指标。 | 作为 P1/P2 扩展，不阻塞 MVP。 |

首批已确认纳入的 8 张表模型：

| 类型 | 表中文名 | 中台目标表 |
|---|---|---|
| 产品绩效事实 | 每日产品净值收益表 | `T_DWS_AS_PD_INFO_I` |
| 产品绩效事实 | 每日产品及基准收益率表 | `T_DWS_AS_PD_BM_YIELD_I` |
| 资产类绩效事实 | 每日产品资产类持仓收益表 | `T_DWS_AS_PD_AC_INFO_I` |
| 证券绩效事实 | 每日证券持仓收益表 | `T_DWS_AS_PD_SEC_INFO_I` |
| 证券绩效事实 | 每日证券收益率表 | `T_DWS_AS_PD_SEC_YIELD_I` |
| 产品主数据 | 产品基本信息 | `T_DWD_PD_BSCINFO_A` / `T_DWS_PD_BSCINFO_A` |
| 日期维度 | 日维表 | `T_DWD_GL_DAY` |
| 证券主数据 | 证券内码表 | `T_DWD_IN_SEC_A` |

暂不纳入首批主题域边界：

1. 完整指标平台配置、指标发布和指标消费。
2. 全量绩效风控 3.0 表迁移。
3. 全市场行情、债券市场指标、股票市场指标的完整模型治理。
4. 交易流水、估值核算、组合树、虚拟组合、汇总组合等复杂主数据扩展。
5. 风险因子、归因算法、绩效计算引擎和风控规则执行。

## 4. 非目标范围

1. 不做数据集成、ETL 编排、任务调度和运维监控。
2. 不自动执行数据库迁移脚本；只输出 DDL 草案并要求人工确认。
3. 不做完整指标平台；P0 只允许在模型内记录必要的指标/维度描述。
4. 不做完整数据质量平台；P0 的规范检查只覆盖模型设计时的元数据规则。
5. 不做完整血缘引擎和资产地图。
6. 不做 BI 数据集、仪表板、分析门户和语义消费层。
7. 不做企业级权限中台；P0 只定义满足评审流的轻量角色边界。
8. 不做 AI 自动建模或自然语言问数。
9. 不做跨公司商业化、多租户计费和外部客户交付能力。

## 6. 验收标准

### 6.1 主题域与逻辑模型

```gherkin
假如数据架构师需要为一个新的业务数据模型建模
当他创建模型项目、填写项目边界、主题域边界和负责人，并新增至少一个业务对象、逻辑实体和逻辑字段
那么系统应保存完整模型草稿，并能展示该模型项目下的主题域、业务对象和逻辑模型结构
```

### 6.2 物理模型映射

```gherkin
假如数仓开发工程师正在为已定义的逻辑模型设计物理表
当他新增物理表、物理字段并建立逻辑字段映射
那么系统应展示字段级映射覆盖率，并标出未映射或重复映射的字段
```

```gherkin
假如模型准备提交评审
当整体字段映射覆盖率低于 80%
那么系统应阻断提交，并提示未达到映射覆盖率及格线
```

```gherkin
假如模型准备发布
当整体字段映射覆盖率低于 85%，或 P0 必填逻辑字段映射覆盖率低于 95%
那么系统应阻断发布，并提示未达到发布门禁
```

### 6.3 规范检查

```gherkin
假如模型存在 P0 级规范问题
当用户提交模型评审
那么系统应阻断提交，并展示命名、分层、字段类型、注释、敏感标记、待分级状态或映射完整性问题
```

```gherkin
假如模型只存在警告级规范问题
当用户提交模型评审
那么系统允许提交，并在评审页保留警告列表和处理说明入口
```

### 6.4 模型评审

```gherkin
假如数仓开发工程师提交一次模型变更
当数据架构师进入评审页
那么系统应展示本次变更 diff、规范检查结果、评论区和通过/驳回操作
```

```gherkin
假如评审被驳回
当提交人修改模型后再次提交
那么系统应保留上一轮评审意见，并展示新一轮变更和检查结果
```

### 6.5 发布与交付资产

```gherkin
假如模型评审已通过
当负责人发布模型
那么系统应生成不可变发布快照、版本号、发布说明、模型文档、字段映射表和 DDL 草案
```

```gherkin
假如用户导出 DDL 草案
当导出完成
那么系统只提供 SQL 草稿文件或文本，不连接数据库，也不执行任何建表或改表操作
```

### 6.6 版本追溯

```gherkin
假如模型项目已有多个发布版本
当用户查看历史版本
那么系统应展示版本号、发布时间、发布人、发布说明和对应发布快照
```

## 7. PRD 输入

### 7.1 PRD 建议标题

企业数据模型协作工作台 MVP

### 7.2 问题陈述

公司内部数据架构师和数仓开发工程师缺少统一的模型协作工作台。逻辑模型常停留在文档中，物理表落地依赖人工沟通，导致字段口径漂移、命名和类型不一致、敏感字段标识遗漏、评审难以聚焦变更、发布后缺少可追溯模型资产。

### 7.3 解决方案

建设一个面向内部用户的企业数据模型协作工作台，以模型项目为顶层协作单元，以主题域为业务边界组织业务对象和逻辑模型，让数仓开发基于逻辑模型维护物理模型映射，并通过规范检查、评审、发布快照和交付资产导出形成端到端闭环。

| FR-004 | 支持映射覆盖率展示、未映射字段提示和覆盖率及格线门禁。 | P0 |
| FR-005 | 支持命名、分层、字段类型、注释、必填元数据、敏感标记、待分级状态和映射完整性检查。 | P0 |
| FR-006 | 支持单一数据架构师审批，包括提交评审、评论、驳回、通过和再次提交。 | P0 |

### 7.5 非功能需求初稿

| 类别 | 需求 | 初始目标 |
|---|---|---|
| 性能 | 单模型项目加载和规则检查可支撑中等规模模型。 | 50 个逻辑实体、200 张物理表、5000 个字段内，常规查询和检查在 5 秒内返回。 |
| 可追溯 | 发布后的版本不可变，评审和发布记录可追溯。 | 任一发布版本可查看快照、发布说明和提交人。 |
| 安全 | DDL 草案不得自动执行数据库变更。 | 系统不连接生产库执行建表或改表。 |
| 可配置 | 规则需要能适配不同团队的命名和分层规范。 | P0 支持小规模规则配置，复杂规则引擎后置。 |
| 可用性 | 架构师和开发可以围绕同一次变更协作。 | 评审页聚合 diff、检查结果、评论和结论。 |
| 数据库适配 | P0 优先生成 PostgreSQL DDL 草案。 | Oracle / Hive / Doris / ClickHouse / StarRocks 等后置。 |

| 导入链路 | 导入既有表结构 -> 生成物理模型草稿 -> 建立逻辑字段映射 -> 检查覆盖率。 |

### 7.8 AI / 人工审查点

| 范围 | Agent 权限 | 人工审查 |
|---|---|---|
| DDL 草案 | 仅生成草案 | 必须人工确认后才能进入真实数据库变更。 |
| 数据库迁移脚本 | 仅生成模板 | 必须人工审查。 |
| 认证 / 授权 | 仅生成草案 | 评审流角色和权限落地前必须人工确认。 |
| 原生 SQL | 仅生成草案 | 涉及执行计划或生产库操作时必须人工确认。 |
| 敏感字段正式分级 | P0 仅保留待分级状态 | 正式分级体系后置，后续需数据治理负责人确认。 |

### 7.9 成功指标建议

| 指标 | 目标方向 |
|---|---|
| 模型交接时间 | 首批试点模型项目的模型交接整理时间下降。 |
| 规范问题前置率 | 命名、注释、类型、敏感标记、待分级状态等问题在评审前被发现。 |
| 模型资产覆盖率 | 首批试点模型项目内关键表、字段和映射关系被纳入模型资产。 |
| 映射覆盖率 | 提交评审时整体字段映射覆盖率不低于 80%，发布时整体字段映射覆盖率不低于 85%，P0 必填逻辑字段映射覆盖率不低于 95%。 |
| 评审效率 | 评审人可通过 diff 和检查结果快速判断变更范围。 |
| 资产复用 | 发布后的模型文档、字段映射表和 DDL 草案被开发流程复用。 |
