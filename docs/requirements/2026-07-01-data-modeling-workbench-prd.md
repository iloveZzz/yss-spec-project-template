# 产品需求文档：企业数据模型协作工作台 MVP

## 问题陈述

公司内部数据架构师和数仓开发工程师缺少统一的企业数据模型协作工作台。当前逻辑模型、物理表设计、字段标准和开发交接材料容易散落在文档、Excel 和已有表结构中，导致逻辑字段落地时发生口径漂移、命名不一致、类型不一致、注释缺失、待分级字段不可见，以及评审时难以快速判断本次变更范围。

本 PRD 的第一版要求工作台能够承载多个业务数据模型项目，其中“绩效风控指标模型复用”是首个 P0 试点模型项目，首批子域为“产品绩效收益模型”。目标不是一次性建设完整数据中台，而是先让该试点项目下的 8 张已确认试点表完成从既有表导入、逻辑模型定义、物理模型映射、规范检查、模型评审、版本发布到交付资产导出的闭环。

## 解决方案

建设一个面向内部数据架构师和数仓开发工程师的企业数据模型协作工作台。工作台以模型项目为顶层协作单元，支持多个业务数据模型项目并行管理；每个模型项目再以主题域和子域组织业务对象、逻辑模型和物理模型。数仓开发工程师可以在项目内导入已有表结构作为物理模型草稿，并建立逻辑字段到物理字段的模型映射。

系统在提交评审和发布前执行规范检查和覆盖率门禁：提交评审时整体字段映射覆盖率不低于 80%，发布时整体字段映射覆盖率不低于 85%，P0 必填字段映射覆盖率不低于 95%。整体覆盖率按范围内有效映射逻辑字段数 / 范围内逻辑字段总数计算，P0 覆盖率按 P0 必填逻辑字段计算。通过单一数据架构师审批后，系统生成发布快照，并以异步任务导出 SQL 文本、Markdown 模型文档和 Excel 字段清单。P0 优先生成 PostgreSQL DDL 草案，不自动执行数据库迁移。

## 用户故事

1. 作为数据架构师，我希望在工作台中管理多个模型项目，以便不同业务数据模型可以独立规划、评审、发布和追溯。
2. 作为数据架构师，我希望创建“绩效风控指标模型复用”模型项目并声明项目边界，以便把它作为多个业务数据模型项目中的首个试点。
3. 作为数据架构师，我希望在模型项目下创建“绩效风控”主题域和“产品绩效收益模型”首批子域，以便先验证产品、资产类、证券、日期和证券主数据相关模型闭环。
4. 作为数据架构师，我希望登记模型项目负责人、主题域负责人和评审人，以便模型变更有清晰责任人。
5. 作为数据架构师，我希望维护模型项目内的主题域、子域和业务对象列表，以便逻辑模型按业务语义组织，而不是直接按表名堆叠。
6. 作为数据架构师，我希望定义逻辑实体、逻辑字段、字段语义和必填属性，以便形成可评审的逻辑模型。
7. 作为数据架构师，我希望为逻辑字段标记敏感标记和待分级状态，以便正式敏感分级体系后置时仍能保留治理入口。
8. 作为数据架构师，我希望复用公司已有字段标准、命名词典和码表 Excel，以便减少重复定义和口径漂移。
9. 作为数仓开发工程师，我希望导入已有表结构作为物理模型草稿，以便降低存量表补录成本。
10. 作为数仓开发工程师，我希望导入 Excel 字段清单后看到导入预览和校验结果，以便在入库前发现缺列、重复字段或类型不合法问题。
11. 作为数仓开发工程师，我希望维护物理表、字段、分层、主键、分区和 PostgreSQL 类型，以便形成可落地的物理模型。
12. 作为数仓开发工程师，我希望建立逻辑字段到物理字段的模型映射，以便让架构设计和落表方案形成可追溯关系。
13. 作为数仓开发工程师，我希望看到整体字段映射覆盖率，以便判断是否达到提交评审条件。
14. 作为数仓开发工程师，我希望看到 P0 必填字段映射覆盖率，以便判断是否达到发布条件。
15. 作为数仓开发工程师，我希望系统标出未映射、重复映射或疑似错映射字段，以便快速修正。
16. 作为数仓开发工程师，我希望在提交评审前运行规范检查，以便提前发现命名、分层、类型、注释、敏感标记、待分级状态和映射完整性问题。
17. 作为数据架构师，我希望配置哪些规范问题是阻断项、哪些是警告项，以便匹配公司内部建模规则。
18. 作为数仓开发工程师，我希望当整体映射覆盖率低于 80% 时无法提交评审，以便避免评审过早进入无效状态。
19. 作为数据架构师，我希望在评审页看到变更 diff、规范检查结果、覆盖率和评论区，以便集中判断本次变更是否可通过。
20. 作为数据架构师，我希望可以通过或驳回一次模型评审，以便形成明确评审结论。
21. 作为数仓开发工程师，我希望评审驳回后可以修改模型并再次提交，以便保留完整修订过程。
22. 作为数据架构师，我希望发布前系统检查整体覆盖率不低于 85%、P0 必填字段覆盖率不低于 95%，以便发布模型具备基本可交付质量。
23. 作为数仓负责人，我希望发布通过评审的模型版本并生成不可变发布快照，以便后续追溯版本。
24. 作为数仓开发工程师，我希望发布后导出 PostgreSQL DDL 草案，以便进入人工确认和开发交接。
25. 作为数仓开发工程师，我希望发布后导出 Markdown 模型文档，以便向架构、开发和治理团队说明模型内容。
26. 作为数仓开发工程师，我希望发布后导出 Excel 字段清单，以便复用现有表模型协作习惯。
27. 作为数据架构师，我希望查看历史版本、发布人、发布时间和发布说明，以便定位口径变化。
28. 作为数据架构师，我希望首批 8 张表的字段、映射和检查结果纳入模型资产，以便衡量模型资产覆盖率。
29. 作为产品负责人，我希望用模型交接时间、规范问题前置率和模型资产覆盖率衡量 MVP 成功，以便判断是否继续扩展到 P1。

## 功能需求

| ID | 需求 | 优先级 | 备注 |
|---|---|---|---|
| FR-001 | 支持模型项目、负责人、项目边界、主题域、业务边界和业务对象管理。 | P0 | 模型项目是工作台顶层协作单元；主题域必须有明确边界，不能作为无限扩展的资产桶。 |
| FR-002 | 支持“绩效风控指标模型复用”试点模型项目，以及项目内“绩效风控”主题域和“产品绩效收益模型”首批子域。 | P0 | 首批纳入已确认的 8 张表模型。 |
| FR-003 | 支持逻辑实体、逻辑字段、字段属性、敏感标记、待分级状态和关系说明。 | P0 | 正式敏感分级体系后置。 |
| FR-004 | 支持导入已有表结构作为物理模型草稿。 | P0 | P0 优先支持 Excel 字段清单导入。 |
| FR-005 | 支持物理表、物理字段、分层、主键、分区和 PostgreSQL 类型维护。 | P0 | 其他数据库类型后置。 |
| FR-006 | 支持逻辑字段和物理字段之间的模型映射。 | P0 | 用于覆盖率、diff、评审和交付资产生成。 |
| FR-007 | 支持整体字段映射覆盖率和 P0 必填字段映射覆盖率展示。 | P0 | 提交评审整体覆盖率门槛 80%；发布整体覆盖率门槛 85%；P0 必填字段发布门槛 95%。 |
| FR-008 | 支持未映射字段、重复映射字段和低覆盖率提示。 | P0 | 低于门槛时阻断对应流程。 |
| FR-009 | 支持命名、分层、字段类型、注释、必填元数据、敏感标记、待分级状态和映射完整性检查。 | P0 | 检查结果分为阻断和警告。 |
| FR-010 | 支持单一数据架构师审批。 | P0 | 包含提交评审、评论、驳回、通过、再次提交。 |
| FR-011 | 支持模型变更 diff。 | P0 | 至少覆盖实体、字段、物理表、映射和规则检查结果。 |
| FR-012 | 支持发布版本、发布快照、发布说明和历史版本查看。 | P0 | 发布快照不可变。 |
| FR-013 | 支持导出 PostgreSQL DDL 草案。 | P0 | 只导出草案，不自动执行。 |
| FR-014 | 支持导出 Markdown 模型文档。 | P0 | 用于开发和评审交接。 |
| FR-015 | 支持导出 Excel 字段清单。 | P0 | 对齐现有公司标准资产形态和表模型协作方式。 |
| FR-016 | 支持公司级字段标准、命名词典和码表 Excel 的复用入口。 | P1 | P0 可预留导入入口，完整标准库治理后置。 |
| FR-017 | 支持指标与维度增强。 | P1 | 原子指标、派生指标、业务限定、维度与模型关联后置。 |
| FR-018 | 支持发布影响分析和研发治理衔接。 | P1 | 与数据开发、发布包、数据地图、质量规则或工单系统对接后置。 |
| FR-019 | 支持统一权限状态和动作可用性展示。 | P0 | 页面无查看权限时不泄露数据；关键流程动作禁用并展示原因；入口型动作可隐藏；越权调用由后端拒绝。 |
| FR-020 | 支持取消、返回、重试和草稿保留规则。 | P0 | 覆盖导入向导、字段 / 映射抽屉、提交评审、评审驳回、发布、异步导出和草稿冲突。 |
| FR-021 | 支持模型项目、字段映射和版本历史列表的分页、筛选和排序。 | P0 | 默认排序和可排序字段进入 OpenAPI Draft 细化；至少覆盖最近更新时间、状态、覆盖率、阻断问题数和发布时间。 |

## 非功能需求

| 类别 | 需求 | 目标 |
|---|---|---|
| 性能 | 单模型项目加载和规则检查可支撑中等规模模型。 | 50 个逻辑实体、200 张物理表、5000 个字段内，常规查询和检查在 5 秒内返回。 |
| 可追溯 | 发布后的版本不可变，评审和发布记录可追溯。 | 任一发布版本可查看快照、发布说明、发布人和发布时间。 |
| 安全 | DDL 草案不得自动执行数据库变更。 | 系统不连接生产库执行建表或改表。 |
| 数据库适配 | P0 优先生成 PostgreSQL DDL 草案。 | Oracle、Hive、Doris、ClickHouse、StarRocks 等后置。 |
| 可配置 | 规则需要能适配不同团队的命名和分层规范。 | P0 支持小规模规则配置，复杂规则引擎后置。 |
| 可用性 | 架构师和开发可以围绕同一次变更协作。 | 评审页聚合 diff、检查结果、覆盖率、评论和结论。 |
| 兼容性 | 公司级字段标准、命名词典和码表资产现有形态为 Excel。 | P0 导入链路需要优先兼容 Excel。 |
| 权限一致性 | 权限状态需要在前端展示和后端拒绝之间保持一致。 | 页面无查看权限不返回受限数据；动作权限返回 enabled 和 disabledReason；后端对越权调用返回 403。 |

## 验收标准

```gherkin
假如数据架构师创建“绩效风控指标模型复用”模型项目
当他填写项目负责人、项目边界、主题域边界，并创建“产品绩效收益模型”子域
那么系统应保存模型项目、主题域和子域，并展示该模型项目的明确边界
```

```gherkin
假如数仓开发工程师导入 Excel 字段清单
当系统完成导入预览
那么系统应展示将生成的物理表、物理字段、数据类型和校验问题
```

```gherkin
假如数仓开发工程师上传现有公司 Excel 字段清单
当系统能够把表头别名映射到模型工具标准模板列
那么系统应进入导入预览，并展示标准列、原始列、错误和警告
```

```gherkin
假如数仓开发工程师上传的 Excel 缺少标准模板必填列，或目标表不在首批 8 张表范围内
当系统解析导入文件
那么系统应阻断生成物理模型草稿，并展示缺失列、越界表和标准模板下载入口
```

```gherkin
假如数据架构师定义逻辑实体和逻辑字段
当数仓开发工程师建立逻辑字段到物理字段的模型映射
那么系统应展示整体字段映射覆盖率、P0 必填字段映射覆盖率和未映射字段列表
```

```gherkin
假如系统计算字段映射覆盖率
当逻辑字段存在有效物理字段映射
那么整体覆盖率应按“范围内有效映射逻辑字段数 / 范围内逻辑字段总数”计算，P0 覆盖率应按 P0 必填逻辑字段单独计算
```

```gherkin
假如模型准备提交评审
当整体字段映射覆盖率低于 80%
那么系统应阻断提交，并提示未达到提交评审门禁
```

```gherkin
假如模型准备发布
当整体字段映射覆盖率低于 85%，或 P0 必填逻辑字段映射覆盖率低于 95%
那么系统应阻断发布，并提示未达到发布门禁
```

```gherkin
假如模型存在阻断级规范问题
当用户提交模型评审
那么系统应阻断提交，并展示命名、分层、字段类型、注释、敏感标记、待分级状态或映射完整性问题
```

```gherkin
假如数仓开发工程师提交一次模型变更
当数据架构师进入评审页
那么系统应展示变更 diff、规范检查结果、映射覆盖率、评论区和通过/驳回操作
```

```gherkin
假如模型评审已通过
当负责人发布模型
那么系统应生成不可变发布快照、版本号、发布说明、模型文档、字段映射表和 PostgreSQL DDL 草案
```

```gherkin
假如用户导出 DDL 草案
当导出完成
那么系统只提供 SQL 草稿文件或文本，不连接数据库，也不执行任何建表或改表操作
```

```gherkin
假如用户导出 PostgreSQL DDL 草案、Markdown 模型文档或 Excel 字段清单
当用户点击导出
那么系统应创建异步导出任务，并展示 queued、running、succeeded、failed 或 expired 状态
```

```gherkin
假如导出任务失败
当用户点击重试
那么系统应基于原导出参数创建新的导出任务，并保留失败原因用于追溯
```

```gherkin
假如模型项目已有多个发布版本
当用户查看历史版本
那么系统应展示版本号、发布时间、发布人、发布说明和对应发布快照
```

```gherkin
假如非评审人打开模型评审页
当系统返回该用户没有评审权限
那么系统应允许其查看有权查看的 diff、检查结果和评论，并禁用通过 / 驳回按钮且展示禁用原因
```

```gherkin
假如用户打开已发布版本
当用户尝试编辑字段、映射或发布说明
那么系统应进入只读态，禁用编辑、保存和发布动作，并提供新建草稿变更入口
```

```gherkin
假如用户在字段编辑抽屉或映射编辑抽屉中存在未保存修改
当用户点击取消、关闭或离开页面
那么系统应提示保存草稿、放弃修改或继续编辑，避免误丢修改
```

```gherkin
假如用户的模型草稿发生版本冲突
当用户保存字段或映射
那么系统应阻断覆盖他人修改，并提示刷新到最新草稿或放弃本地修改
```

| 表结构导入 | Excel 字段清单上传、导入预览、导入校验、导入结果保存。 |
| 规范检查 | 规则配置、检查执行、检查结果、阻断和警告分类。 |
| 模型评审 | 提交、评论、通过、驳回、再次提交、评审记录查询。 |
| 版本发布 | 发布、发布快照、历史版本、发布说明和异步交付资产导出。 |
| 权限动作 | 页面查看权限、动作可见性、动作可用性、禁用原因和越权拒绝。 |
| 查询排序 | 模型项目列表、字段映射列表、版本历史列表的分页、筛选和排序参数。 |

## 实施决策

1. 生命周期阶段进入 Requirements / PRD，后续不得直接进入实现，必须先完成 UI 原型和 OpenAPI Draft。
2. P0 工作台支持多个模型项目，首个试点模型项目为“绩效风控指标模型复用”，项目内首批子域为“产品绩效收益模型”。
3. 首批采用已确认的 8 张表，不再压缩为 3 到 5 张表，也不扩展到全量指标平台模型。
4. P0 优先支持 PostgreSQL DDL 草案生成，其他数据库适配后置。
5. P0 评审采用单一数据架构师审批，不引入多级审批。
6. P0 支持 Excel 字段清单导入已有表结构，数据库元数据直连采集后置。
7. 公司级字段标准、命名词典和码表资产现有形态为 Excel，P0 以 Excel 复用为主。
8. 正式敏感分级体系后置，P0 只保留敏感标记和待分级状态。
9. 覆盖率门禁为：提交评审整体字段映射覆盖率 80%，发布整体字段映射覆盖率 85%，P0 必填字段映射覆盖率 95%。
10. 发布产物必须包含 SQL 文本、Markdown 模型文档和 Excel 字段清单。
11. DDL 只作为草案导出，不自动执行数据库迁移。
12. 权限采用“页面无权不泄露数据、关键流程动作禁用并给原因、入口型动作可隐藏、后端兜底拒绝”的统一策略。
13. Excel 导入以模型工具标准模板为准，同时兼容现有公司 Excel 表头别名；无法识别到标准列时阻断并要求人工整理。
14. SQL、Markdown、Excel 三类交付资产统一使用异步导出任务，状态为 queued、running、succeeded、failed、expired。
15. P0 不做草稿冲突自动合并；发生版本冲突时用户刷新最新草稿或放弃本地修改。
16. 查询类接口默认支持分页、筛选和排序；OpenAPI Draft 需要统一 `page`、`size`、`sort`、`direction` 参数命名。

## 测试决策

主要测试 seam：

1. 领域行为 seam：验证映射覆盖率计算、P0 必填字段覆盖率计算、阻断/警告规则判定、发布快照不可变。
2. API / 契约 seam：验证模型草稿保存、Excel 导入预览、提交评审、评审结论、发布和导出接口契约。
3. 前端组件 seam：验证模型项目列表、模型项目详情、模型编辑表格、映射覆盖提示、评审 diff、导出入口和错误状态。
4. E2E 关键路径 seam：验证“创建模型项目 -> 定义主题域边界 -> 导入已有表 -> 定义逻辑模型 -> 建立物理映射 -> 规范检查 -> 提交评审 -> 架构师通过 -> 发布 -> 导出三类资产”的端到端流程。
5. 权限状态 seam：验证无查看权限不返回受限数据、非评审人审批按钮禁用、越权接口调用返回 403。
6. 导入模板 seam：验证标准模板、现有 Excel 表头别名、缺失必填列、目标表越界和重复字段。
7. 异步导出 seam：验证 queued、running、succeeded、failed、expired 状态和失败重试创建新任务。
8. 流程恢复 seam：验证抽屉脏表单离开确认、导入取消不生成草稿、发布失败保留发布说明、并发冲突不覆盖他人修改。
9. 查询排序 seam：验证模型项目、字段映射和版本历史列表的分页、筛选、默认排序和显式排序参数。

测试原则：

1. 优先测试外部可观察行为，不测试内部实现细节。
2. 覆盖率门禁和 DDL 草案安全边界必须有自动化测试。
3. 发布快照不可变必须作为领域行为测试。
4. Excel 导入需要覆盖标准模板、表头别名、缺列、重复字段、空字段名、不支持的数据类型和超出首批模型项目 / 主题域边界的输入。
5. OpenAPI Draft 完成后需要补充契约测试。

## AI / 人工审查点

| 范围 | Agent 权限 | 是否需要人工审查 | 备注 |
|---|---|---|---|
| DDL 草案 | 仅生成草案 | 是 | 必须人工确认后才能进入真实数据库变更。 |
| 数据库迁移脚本 | 仅生成模板 | 是 | 不得由 Agent 自动执行。 |
| 认证 / 授权 | 仅生成草案 | 是 | 评审流角色和权限落地前必须人工确认。 |
| 原生 SQL | 仅生成草案 | 是 | 涉及执行计划或生产库操作时必须人工确认。 |
| 敏感字段正式分级 | P0 仅保留待分级状态 | 是 | 正式分级体系后置，后续需数据治理负责人确认。 |
| 公共基础库 API | 仅生成草案 | 是 | 如后续涉及平台公共能力变更，需要人工审查。 |

## 非目标范围

1. 不做数据集成、ETL 编排、任务调度和运维监控。
2. 不自动执行数据库迁移脚本。
3. 不做完整指标平台配置、指标发布和指标消费。
4. 不做全量绩效风控 3.0 表迁移。
5. 不做全市场行情、债券市场指标、股票市场指标的完整模型治理。
6. 不做交易流水、估值核算、组合树、虚拟组合、汇总组合等复杂主数据扩展。
7. 不做风险因子、归因算法、绩效计算引擎和风控规则执行。
8. 不做完整数据质量平台；P0 的规范检查只覆盖模型设计时的元数据规则。
9. 不做完整血缘引擎和资产地图。
10. 不做 BI 数据集、仪表板、分析门户和语义消费层。
11. 不做企业级权限中台。
12. 不做 AI 自动建模或自然语言问数。
13. 不做跨公司商业化、多租户计费和外部客户交付能力。

## 风险

| 风险 | 影响 | 缓解方式 |
|---|---|---|
| 8 张表一次纳入后字段量较大 | 可能拉长首批建模和校验周期。 | 仍以同一模型项目闭环为目标，允许按表分批导入和评审，但 PRD 范围保持 8 张表。 |
| Excel 标准资产结构不统一 | 导入规则可能复杂。 | P0 以模型工具标准模板为准，兼容现有 Excel 表头别名，无法识别时要求人工整理。 |
| 覆盖率门禁较低可能放过模型缺口 | 发布资产质量可能受影响。 | P0 必填字段设置 95% 门槛，并要求规范检查展示未映射字段。 |
| 正式敏感分级后置 | 早期不能完整支撑数据安全治理。 | P0 保留敏感标记和待分级状态，后续由数据治理负责人确认分级体系。 |
| PostgreSQL 优先可能与历史 Oracle 类型存在差异 | DDL 草案转换可能存在类型映射争议。 | 类型映射需在 OpenAPI / 设计阶段形成明确规则，DDL 只做草案不自动执行。 |

冻结范围：

1. 工作台顶层支持多个模型项目，首个 P0 试点项目为“绩效风控指标模型复用”。
2. 试点项目固定为“绩效风控”主题域、“产品绩效收益模型”首批子域和已确认 8 张 P0 表。
3. MVP 闭环固定为：项目 / 主题域边界、Excel 导入、逻辑模型、物理模型、字段映射、覆盖率、规范检查、单一架构师评审、版本发布、三类异步导出。
4. 覆盖率门禁固定为：提交评审整体覆盖率 80%，发布整体覆盖率 85%，P0 必填字段覆盖率 95%。
5. 权限策略固定为：页面无权不泄露数据、关键流程动作禁用并给原因、入口型动作可隐藏、后端兜底拒绝。
6. Excel 导入策略固定为：模型工具标准模板为规范模板，兼容现有公司 Excel 表头别名，无法识别时阻断并要求人工整理。
7. 导出策略固定为：SQL、Markdown、Excel 三类资产统一异步导出。
8. 数据库边界固定为：P0 只生成 PostgreSQL DDL 草案，不连接数据库，不自动执行迁移。

冻结后变更控制：

1. 任何新增 P0 业务范围、试点表、审批层级、数据库直连、DDL 执行、权限模型变更，必须回到 PRD 校准和原型评审。
2. 任何 OpenAPI Draft 中发现的字段、错误结构、排序参数或 schema 命名补充，只要不改变 P0 行为，可作为契约细化直接记录到 OpenAPI Draft。
3. 任何涉及认证 / 授权、数据库迁移、原生 SQL、公共基础库 API 的实现方案，只能形成草案并标记人工审查。
