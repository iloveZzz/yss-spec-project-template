# grill-with-docs 最佳实践

本文补充 [grill-with-docs 使用手册](./grill-with-docs-guide.md)，用于约束长期使用 `grill-with-docs` 时的工作习惯、产物边界和常见反模式。

## 1. 核心原则

### 1.1 追问优先于生成

`grill-with-docs` 的目标不是马上写一份漂亮文档，而是把需求里的模糊点问出来。

好的使用方式：

```text
请先连续追问，不要直接生成 PRD。
```

不好的使用方式：

```text
我有个想法，直接帮我生成完整 PRD。
```

如果 AI 开始替你补全关键业务规则，要把它拉回来：

```text
这个规则不要猜，请作为待确认问题列出。
```

### 1.2 术语稳定后立刻沉淀

当一个业务概念已经确定，就写入 `CONTEXT.md`。不要等整轮讨论结束再批量整理，因为批量整理容易把“已确认”和“仍在猜”混在一起。

写入标准：

- 是本项目特有的业务语言。
- 定义能用一两句话说清。
- 有同义词或容易误用的词需要规避。
- 后续 PRD、OpenAPI、代码、测试都会复用。

不写入：

- 通用技术词。
- 类名、表名、字段名。
- 临时讨论结论。
- 尚未确认的猜测。

### 1.3 ADR 要少而准

ADR 不是会议纪要，也不是方案全文。它只记录难以回滚、非显而易见、经过真实取舍的决定。

在 grill 中出现这类问题时才考虑 ADR：

- 将来改掉成本很高。
- 未来维护者会疑惑为什么这样做。
- 当时有多个可行选择。

如果只是“常规实现方式”，不要写 ADR。

### 1.4 不要把 grill 当实现计划

`grill-with-docs` 产出的不是开发任务清单。它负责澄清：

- 谁用。
- 做什么。
- 为什么做。
- 不做什么。
- 业务词怎么定义。
- 哪些决定需要记录。

实施计划应交给：

- `to-prd`
- `writing-plans`
- `OpenSpec / Comet`
- `yss-router`

## 2. 推荐追问维度

### 2.1 用户与角色

必须问清：

- 谁是主要用户。
- 谁配置，谁审批，谁查看结果。
- 管理员和普通业务用户的边界。
- 实施人员是否有特殊操作入口。

示例：

```text
模型发布是建模人员自己发布，还是需要平台管理员审批？
```

### 2.2 业务对象和状态

必须问清：

- 核心对象是什么。
- 对象有哪些状态。
- 状态能否回退。
- 状态变化是否产生审计记录。

示例：

```text
草稿版本发布后，是生成一个新的发布版本，还是直接把草稿状态改为已发布？
```

### 2.3 边界和非目标

必须问清：

- 第一版做什么。
- 第一版明确不做什么。
- 哪些能力只是竞品参考，不进入 MVP。
- 哪些能力必须人工确认。

示例：

```text
模型发布第一版是否包含审批流、回滚、血缘影响分析和通知？
```

### 2.4 异常和失败路径

必须问清：

- 校验失败怎么提示。
- 并发编辑怎么处理。
- 下游引用时发布失败怎么办。
- 权限不足、网络失败、数据不存在如何反馈。

示例：

```text
如果发布时字段校验失败，用户需要看到字段级错误，还是只看到整体失败原因？
```

### 2.5 验收和可验证性

必须问清：

- 什么结果算完成。
- 哪些场景必须自动化测试。
- 哪些场景必须人工验收。
- 哪些日志、审计、实施记录需要保留。

示例：

```text
发布成功后，如何证明下游只能引用发布版本，不能引用草稿版本？
```

## 3. 与产品全生命周期结合

### 3.1 竞品分析之后

竞品分析给出“行业常见能力”和“差异化机会”，grill 负责把它变成你的产品边界。

推荐输入：

```text
使用 grill-with-docs，基于 docs/discovery/reports/model-management-competitive-matrix.md，
追问哪些竞品能力进入 MVP，哪些作为非目标。
```

输出应能支撑 discovery 和 PRD。

### 3.2 PRD 之前

PRD 之前使用 grill，可以降低“AI 代替产品经理做决定”的风险。

推荐输出：

- 已确认用户故事。
- 非目标范围。
- 关键验收标准。
- OpenAPI 影响初判。
- 待人工确认问题。

### 3.3 OpenSpec / Comet 之前

OpenSpec / Comet 需要清晰 change 目标。如果目标还模糊，先 grill。

进入 OpenSpec 前至少确认：

- change 名称表达一个目标。
- 主要行为和非目标清楚。
- API 影响明确。
- 是否触碰安全红线。
- 是否存在需要 ADR 的取舍。

### 3.4 YSS 实现之前

进入 `yss-router` 前，grill 应该已经解决业务问题，不应再把业务规则留给专项实现技能猜。

进入实现前至少确认：

- 页面或接口服务的业务目的。
- 领域对象和状态。
- 验收标准。
- OpenAPI 影响。
- 前后端职责边界。

## 4. 输出格式建议

每轮 grill 结束时，建议要求 AI 输出：

```markdown
## 本轮结论

### 已确认
- ...

### 待确认
- ...

### 非目标
- ...

### 术语沉淀
- `CONTEXT.md`: ...

### ADR 建议
- 无 / 建议新增 `docs/adr/000N-xxx.md`

### 下一步
- ...
```

如果存在待确认问题，不要直接进入实现。

## 5. 常见反模式

| 反模式 | 风险 | 改法 |
|---|---|---|
| AI 一次问十个问题 | 用户难回答，容易敷衍 | 一次聚焦一个主题，必要时一个关键问题一轮 |
| AI 替业务做决定 | 关键规则来自猜测 | 标为待确认 |
| 术语都写进 CONTEXT.md | 词汇表膨胀失真 | 只写稳定业务语言 |
| ADR 写成大方案 | 后续没人读 | 只记录决定和原因 |
| grill 之后直接开发 | PRD / OpenAPI / change 丢失 | 先进入 PRD 或 OpenSpec/Comet |
| 只问正常路径 | 异常场景上线后暴露 | 必问失败、并发、权限、回滚 |
| 把实现细节写入 glossary | 领域语言污染 | 实现细节放 PRD、设计或 ADR |

## 6. 检查清单

### 开始 grill 前

- [ ] 已说明业务背景。
- [ ] 已提供已有材料路径。
- [ ] 已说明希望澄清的主题。
- [ ] 已授权是否可以更新 `CONTEXT.md`。
- [ ] 已说明是否需要 ADR 建议。

### grill 过程中

- [ ] 不确定的问题标记为待确认。
- [ ] 稳定术语及时写入 `CONTEXT.md`。
- [ ] 冲突术语被指出并解决。
- [ ] 正常路径、异常路径、权限和边界都被问到。
- [ ] 重大取舍按 ADR 三条件判断。

### grill 结束后

- [ ] 已确认范围和非目标。
- [ ] 已列出待确认问题。
- [ ] `CONTEXT.md` 没有实现细节。
- [ ] ADR 只记录必要决定。
- [ ] 已明确下一步是 discovery、PRD、OpenAPI、OpenSpec/Comet 还是实现路由。

## 7. 一句话原则

```text
grill-with-docs 的成果不是“问了很多问题”，而是让后续 PRD、规格和代码不再靠猜。
```
