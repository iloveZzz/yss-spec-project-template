# 发现阶段 (Discovery) — 市场分析 · 竞品分析 · 用户洞察

> **Phase 0 — 需求分析的前置输入。**

---

## 工作流

Discovery 不是固定排在 Ideation 后面的单向步骤。它和机会构想共同组成“机会探索环”，用于把事实、假设和 MVP 边界来回校准：

```
市场 / 竞品 / 用户事实 <-> 机会构想假设 <-> MVP 边界
```

进入 PRD 前必须收敛为：

- 用户是谁。
- 痛点是什么。
- 为什么现在做。
- 第一版做什么。
- 明确不做什么。
- 成功标准是什么。
- 产品功能建设方向是什么。
- 后续可能影响哪些 UI、API、数据、权限、架构和安全门禁。

常见入口：

- 只有模糊想法：先机会构想，再用市场、竞品和用户洞察验证。
- 已有行业、竞品或用户材料：先用 `competitive-intelligence` 整理 Discovery 事实，再生成候选方案。
- 明确 Bug 或小调整：不走完整 Discovery，直接进入 hotfix / tweak。

---

## 市场分析

### 数据源
- 行业报告、上市公司财报、政府数据、投资机构研报

### 产出物
- 市场总览报告 / 规模与增长预测 / 趋势洞察

---

## 竞品分析

推荐优先使用 `competitive-intelligence`，把公开证据、事实 / 推断、置信度和 PRD 输入分开记录。竞品分析结论只能作为 Discovery / PRD 输入，不能直接冻结为需求。

### 分析维度
产品功能 / 用户体验 / 技术能力 / 商业模式 / 用户口碑 / 市场表现

### 产出物
- 竞品情报简报 / 竞品功能矩阵 / 竞品深度分析 / 差异化机会清单 / 产品功能指引输入

---

## 用户洞察

### 数据源
- 应用商店评论 / 社交媒体 / 客服社区 / 行业报告

### 产出物
- 用户画像 / 用户旅程地图 / 痛点优先级清单

---

## Discovery 收敛产物

Discovery 的结论用于影响 PRD、功能架构、产品设计、API 和架构阶段，但不替代这些阶段的正式产物。

- 机会说明。
- MVP 范围和非目标范围。
- 产品功能指引 / 能力候选地图。
- 下游影响清单：UI、API、数据、权限、外部系统、发布回滚和安全红线。
