# yss-spec 模板同步 CLI Discovery 收敛文档

> 目标：围绕“CLI 支持持续同步模板演进到已有模板实例仓库”完成需求澄清，输出可直接进入 `to-prd` 的上游结论。

## 1. 先澄清术语

本轮最容易混淆的是“同步”。

- 不是“新项目初始化”。
- 不是“把 GitHub 最新代码重新发布到 npm”。
- 本文默认的“模板同步”是：
  - 让一个已经由 `yss-spec-project-template` 初始化出来的“模板实例仓库”，在后续某个时点对齐到更新后的模板快照版本。

如果后续用户真正想要的是“只要重新发 npm 包即可”，那属于模板发布流程，不属于这里的“实例仓库同步”能力。

## 3. 目标用户与真实场景

| 用户 / 角色 | 真实场景 | 当前痛点 | 影响范围 |
|-------------|----------|----------|----------|
| 模板维护者 | 模板源仓库持续新增流程、文档、脚本、规范后，希望已有项目能跟进 | 只能人工比对目录和文件，容易漏同步、误覆盖 | 模板演进效率、推广成本 |
| 项目初始化负责人 / Tech Lead | 已用 CLI 初始化过项目，后续希望补齐模板新增资产 | 不知道哪些文件该更新、哪些本地改动不能碰 | 项目仓库一致性、升级风险 |
| 研发管理使用者 | 希望项目仓库不断跟进新的流程基线 | 模板升级没有标准路径，导致每个项目分叉越来越大 | 过程规范统一性 |

## 4. 核心问题

- 当前 CLI 只解决“第一次初始化”，没有解决“模板演进后如何让既有实例仓库跟进”。
- 模板实例仓库一旦落地，就会出现本地修改；后续同步不是简单覆盖，而是“受管资产升级 + 本地变更保护 + 冲突可见化”。
- 如果没有模板版本基线，CLI 无法判断：
  - 当前实例仓库是从哪个模板快照创建的
  - 这次准备同步到哪个目标版本
  - 哪些文件是模板新增、模板修改、用户本地修改

## 5. 关键分歧点

### 5.1 “同步”到底同步什么

推荐默认只同步以下“研发管理模板资产”：

- `docs/templates/`
- `docs/process/`
- `docs/agents/`
- `docs/user-guide/` 中明确标记为模板资产的部分
- `AGENTS.md`
- `CONTEXT.md` 中模板占位初始化后的公共段落
- 辅助脚本（如 `scripts/verify-template`、`scripts/gitworks`）

不建议默认自动同步：

- 项目团队已经开始填写的 PRD 正文
- 项目自己的 OpenAPI 草稿
- 已创建的 issue 内容
- 项目特有 README 大段业务说明
- 用户新建的非模板文件

结论：模板同步应是“受管模板资产同步”，不是“整个仓库重置”。

### 5.2 命令形态是否继续沿用 `create-*`

这里有一个真实 trade-off：

- 如果继续沿用 `create-yss-spec`
  - 优点：沿用现有 npm 包，最短路径可做 `npx create-yss-spec@latest sync`
  - 风险：`create-*` 从语义上更像一次性初始化，不像长期运维命令
- 如果升级为稳定 CLI（如 `yss-spec init` / `yss-spec sync`）
  - 优点：命令语义更完整，更适合长期维护
  - 风险：需要额外处理包迁移、别名兼容、文档迁移

建议：

- MVP 先不拆包，仍在当前仓库和当前 npm 包中实现
- 命令层采用过渡方案：
  - `npx create-yss-spec@latest sync`
- 中期再评估稳定命令：
  - `yss-spec init`
  - `yss-spec sync`

### 5.3 同步源来自哪里

候选有三个：

1. 当前执行的 npm 包内置模板快照
2. GitHub 仓库远程拉取最新模板
3. 独立模板清单 / 模板版本服务

建议 MVP 选 1：

- 以当前执行的 npm 包版本作为“目标模板快照版本”
- 好处是发布、复现、回滚最简单
- 用户执行 `npx create-yss-spec@latest sync` 时，自然拿到最新已发布模板

不建议 MVP 直接读 GitHub 最新默认分支：

- 会让“源码最新”与“已验证可发布模板”混在一起
- 难以复现问题
- 不利于回滚

## 6. 推荐的 MVP 边界

| 能力 | 是否进入 MVP | 理由 | 成功信号 |
|------|---------------|------|----------|
| 支持在已有模板实例仓库中执行 `sync` | 是 | 这是本轮核心价值 | 用户能标准化升级模板资产 |
| 为实例仓库写入模板元数据文件 | 是 | 没有基线就无法判断同步关系 | CLI 能识别当前模板版本 |
| 基于受管文件清单执行同步 | 是 | 避免把整个仓库当成可覆盖对象 | 自动更新范围可控 |
| 默认只更新“未被本地修改”的受管文件 | 是 | 保护项目局部自定义 | 无意外覆盖 |
| 对本地已改动的受管文件给出冲突 / 跳过报告 | 是 | 同步必须 fail-closed | 用户知道哪些地方需要人工处理 |
| 支持 `--dry-run` 预览同步计划 | 是 | 同步比初始化更需要预演 | 用户先看变化再执行 |
| 支持 `--from` / `--to` 版本识别或展示 | 是 | 便于理解本次升级跨度 | 输出中能看到版本变更 |
| 支持 `--force` 强制覆盖冲突文件 | 否 | 风险过高，容易伤害项目自定义 | 先人工确认冲突处理 |
| 支持自动三方合并 | 否 | 实现复杂，且容易产出错误合并 | 后续版本再评估 |
| 支持同步项目已填写内容（如 PRD 正文） | 否 | 这些内容已脱离模板资产范畴 | 保持边界清晰 |
| 支持直接从 GitHub 拉取未发布模板 | 否 | 破坏可复现性与发布门禁 | 以后仅作高级模式再评估 |

## 7. 非目标范围

| 非目标 | 排除原因 | 后续观察条件 |
|--------|----------|--------------|
| 整仓库覆盖式升级 | 风险极高，会破坏项目本地资产 | 除非未来引入可靠三方合并和备份机制 |
| 自动解决所有冲突 | 很难保证正确性 | 后续若只针对少量结构化文件可再局部自动化 |
| 同步项目自定义内容 | 已超出模板资产边界 | 仅处理明确受管文件 |
| 在线拉取 GitHub 最新源码直接升级 | 不利于发布治理 | 若未来内部需要 canary 模式再评估 |
| 多模板、多分支、多环境升级平台 | 会显著抬高复杂度 | 等单模板同步路径稳定后再扩展 |

## 8. 推荐的最小产品流

1. 用户在模板实例仓库根目录执行 `npx create-yss-spec@latest sync`
2. CLI 检查当前目录是否存在模板元数据文件，例如 `.yss-template.json`
3. CLI 读取当前实例仓库记录的 `currentTemplateVersion`
4. CLI 以当前 npm 包内模板作为 `targetTemplateVersion`
5. CLI 计算受管文件差异，分类为：
   - `unchanged-managed`：本地未改，可自动更新
   - `modified-managed`：本地已改，需跳过并提示
   - `new-managed`：模板新增，可安全落盘
   - `removed-managed`：模板已删除，MVP 先只提示，不自动删
   - `unmanaged`：不在同步范围内，忽略
6. 若是 `--dry-run`，只输出计划
7. 若正式执行，则：
   - 自动写入 `unchanged-managed`
   - 新增 `new-managed`
   - 跳过 `modified-managed`
   - 报告 `removed-managed`
8. CLI 输出同步摘要和需人工处理的文件清单
9. CLI 更新模板元数据中的当前版本

## 9. 模板元数据建议

建议引入一个实例仓库级元数据文件，例如：

- `.yss-template.json`

建议记录：

- `templateName`
- `templateVersion`
- `templateSource`
- `initializedAt`
- `lastSyncedAt`
- `managedFilesManifestVersion`
- 初始化时采集的核心渲染变量摘要

这个文件的价值是：

- 让 CLI 能判断“这个仓库是不是由本模板初始化出来的”
- 让 CLI 知道“当前仓库上次同步到哪个模板版本”
- 为后续同步报告、回滚和支持问题定位提供依据

## 10. 风险与待澄清

| 风险 / 问题 | 影响 | 建议处理方式 | 截止点 |
|-------------|------|--------------|--------|
| 受管文件范围过大 | 误覆盖项目自定义 | MVP 先只纳入公共模板资产 | PRD 前 |
| 没有模板元数据基线 | 无法可靠识别同步关系 | 把元数据文件纳入 MVP | PRD 前 |
| 模板删除策略不清 | 可能误删用户文件 | MVP 对删除只提示不自动执行 | PRD 前 |
| `create-*` 命令形态长期不适合 sync | UX 心智不清 | PRD 中明确“过渡方案 + 演进路线” | PRD 前 |
| 老项目没有元数据文件 | 无法直接升级 | 需要设计 onboarding / attach 模式 | PRD 前 |

## 11. 对“老项目”的特殊判断

这是一个必须单独说明的边界：

- 对于已经由早期 CLI 初始化、但还没有 `.yss-template.json` 的项目
- CLI 不能直接假设它安全可同步

建议：

- MVP 先只支持“带模板元数据的实例仓库”
- 对老项目提示：
  - 当前仓库缺少模板基线
  - 需要先执行显式 `attach` / `adopt` 流程，或人工确认后再纳入同步

这类“老项目补挂模板基线”的能力，不建议直接塞进第一版同步主路径。

## 12. 成功标准

| 指标 / 结果 | 目标 | 验证方式 |
|-------------|------|----------|
| 同步安全性 | 默认不覆盖本地已改文件 | 行为测试 + 冲突场景测试 |
| 升级可见性 | 用户能在执行前看到同步计划 | `--dry-run` 验证 |
| 同步范围可控 | 仅受管模板资产被自动处理 | 目录快照校验 |
| 版本可追踪 | 每次同步前后都有模板版本标识 | 元数据文件校验 |
| 可回滚性 | 用户至少可通过 Git 看见同步改动 | CLI 输出建议 + 验证文档 |

## 13. 实现位置建议

- 建议继续放在当前仓库中实现，不另起独立仓库。
- 原因：
  - 模板同步比模板初始化更依赖模板权威内容
  - 模板、受管清单、同步规则分仓后更容易漂移
  - 当前仓库本身就是模板源仓库，最适合作为同步规则的单一事实源

目录建议：

- `packages/create-yss-spec/`
  - 新增 `sync` 子命令实现
  - 新增同步相关测试
- `template.manifest.json`
  - 从“初始化复制 / 渲染清单”演进为“初始化 + 同步共用清单”
- 可新增：
  - `docs/implementation/yss-spec-cli-template-sync-routing.md`
  - `docs/requirements/yss-spec-cli-template-sync-prd.md`

## 14. 推荐结论

### 14.1 推荐的产品定义

- 把这件事定义为：
  - “模板实例仓库的受管模板资产同步”
- 不要定义为：
  - “重新生成整个仓库”
  - “从 GitHub 拉最新代码强制覆盖”

### 14.2 推荐的命令方向

- MVP：`npx create-yss-spec@latest sync`
- 中期：演进到 `yss-spec sync`

### 14.3 推荐的同步策略

- 以 npm 已发布版本为同步源
- 以模板元数据作为实例仓库同步基线
- 以受管文件清单控制同步范围
- 默认只更新未被本地修改的受管文件
- 对删除与冲突先提示，不自动处理
