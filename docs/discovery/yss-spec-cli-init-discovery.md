## 2. 目标用户与场景

| 用户 / 角色 | 核心场景 | 当前痛点 | 影响范围 |
|-------------|----------|----------|----------|
| 项目初始化负责人 | 新建一个基于 YSS 研发管理模板的项目仓库 | 手工复制模板、替换占位信息、清理无关文件成本高且容易漏项 | 新项目落地速度、模板一致性 |
| 内部开发者 / Tech Lead | 为新业务线或新客户项目快速落一套规范化管理仓库 | 不清楚哪些文件该改、哪些应保留为模板资产 | 初始化正确率、后续协作成本 |
| 产品 / 架构协作者 | 需要在项目开始时就拿到规范化 PRD / OpenAPI / 架构模板 | 初始仓库内容不稳定，导致后续流程入口不一致 | 生命周期资产可追溯性 |

## 3. 问题与机会

- 问题陈述：当前模板仓库适合作为“权威模板源”，但缺少一个标准化初始化入口，导致新项目往往依赖手工复制、人工删改和经验记忆，容易出现占位信息未替换、无关维护文件被带入、初始化步骤不一致等问题。
- 为什么现在做：模板仓库的流程、文档和技能已经具备复用价值，继续手工初始化会放大团队协作差异和维护成本。
- 差异化机会：把“模板复制 + 变量渲染 + 安全检查 + 初始化选项”收敛到一个 CLI 中，让项目初始化从“会用模板的人才能做”变成“按提示即可完成”。
- 不做的机会：本轮不把它扩展成前后端运行时代码脚手架，也不承担完整平台化工程门户职责。

## 4. MVP 边界

| 能力 | 是否进入 MVP | 理由 | 成功信号 |
|------|---------------|------|----------|
| 提供 `npm create` 风格初始化入口 | 是 | 降低首次使用门槛，符合 npm 生态直觉 | 新用户无需阅读长文档即可启动 |
| 支持交互式收集项目基础信息 | 是 | 替代手工修改模板占位字段 | 生成后无明显未替换占位符 |
| 按规则复制模板文件并排除维护性目录 | 是 | 避免把模板源仓库内部工具和维护目录带进实例仓库 | 生成结果干净、无明显脏资产 |
| 对指定文件做变量渲染，对其余模板资产原样复制 | 是 | 平衡可定制性与实现复杂度 | 关键元信息准确，模板资产仍可复用 |
| 支持目标目录非空保护与 `--force` 覆盖策略 | 是 | 避免误覆盖已有工作目录 | 默认安全、可显式覆盖 |
| 支持 `--dry-run` 预览 | 是 | 便于验证即将生成的文件与变量 | 用户可在落盘前检查结果 |
| 支持可选 `git init` | 是 | 项目初始化后常见下一步 | 新仓库可直接进入 Git checkpoint |
| 自动安装依赖 | 否 | 当前模板仓库不是运行时代码仓库，依赖安装收益低 | 延后到需要前端 / 后端模板时再评估 |
| 远程模板更新 / 同步已有仓库 | 否 | 属于后续维护能力，不是初始化 MVP | 单独作为后续版本能力 |
| 多模板市场 / 多产品线切换 | 否 | 会显著提高配置与发布复杂度 | 先把单模板路径打稳 |

## 12. 澄清结论摘要

### 12.1 目标用户

- 主要用户是“项目初始化负责人”和“内部开发者 / Tech Lead”。
- 产品、架构同学是间接受益者，因为初始化后能更快进入规范化 PRD / OpenAPI / 架构流程。

### 12.2 CLI 名称与调用方式

- 推荐 npm 包名：`create-yss-spec`
- 推荐主调用方式：`npm create yss-spec@latest`
- 推荐兼容方式：`npx create-yss-spec@latest`
- 本轮不优先做 `yss-spec init` 这类多子命令形态；等 CLI 出现第二个核心命令后再考虑升级为 `yss-spec`

### 12.3 初始化输出

- 生成一个新的“模板实例仓库”
- 默认包含当前研发管理模板所需的目录结构、文档模板、流程规范、技能入口资产
- 支持可选 `git init`
- 不默认做依赖安装、远端仓库创建或 CI 对接

### 12.4 交互参数建议

建议 MVP 至少收集以下参数：

| 参数 | 用途 | 是否必填 |
|------|------|----------|
| `projectName` | 项目名称，回填到 `AGENTS.md`、README 等 | 是 |
| `businessDomain` | 业务领域，回填到项目说明 | 是 |
| `teamSize` | 团队规模，回填到项目说明 | 否 |
| `targetDir` | 目标目录 | 是 |
| `initGit` | 是否执行 `git init` | 否 |
| `issueTracker` | GitHub / GitLab / 未定，作为协作默认值 | 否 |
| `includeExampleDocs` | 是否保留示例 / 引导型文档 | 否 |

### 12.5 模板变量与复制策略

- 变量来源：用户交互输入 + CLI 默认值 + 包版本上下文
- 建议把文件分成三类：
  1. `render`：需要替换项目级元信息的文件，例如 `AGENTS.md`、README、部分说明文档
  2. `copy`：原样复制的模板资产，例如大多数 `docs/templates/`、`docs/process/`、技能说明
  3. `exclude`：不进入实例仓库的模板维护文件，例如 npm CLI 源码目录、本地开发配置、仅模板维护使用的目录

### 12.6 版本、更新、覆盖与 dry-run

- 版本选择：MVP 优先依赖 npm 包版本本身，不单独设计复杂的 `--template-version` 协议
- 模板更新：不进入 MVP
- 覆盖策略：默认遇到非空目录即停止，显式传 `--force` 才覆盖
- 预览策略：MVP 必须支持 `--dry-run`

### 12.7 发布策略

- 推荐先面向内部使用，优先私有 npm registry 或受控发包流程
- 若后续明确面向外部社区，再评估 public npm 发布、包名占用和文档开放策略

### 12.8 实现位置建议

- 建议当前仓库明确承载该 CLI 的源码，而不是立刻拆到独立实现仓库。
- 原因：
  - CLI 与模板内容强耦合，分仓会显著增加模板版本与 CLI 版本漂移风险
  - 该工具不是前端 / 后端 runtime 代码，不违背“运行时代码默认独立实现仓库”的主规则
  - 把模板源与初始化工具放在一起，更容易做同版本验证和变更追踪
- 建议目录约定：
  - `packages/create-yss-spec/`：npm CLI 源码
  - `packages/create-yss-spec/tests/`：CLI 行为测试 / 生成快照测试
  - `template.manifest.json` 或等价清单：定义 `render` / `copy` / `exclude` 规则
  - 仓库根目录继续作为权威模板源；如果后续复制规则过于复杂，再评估是否抽出独立 `template/` 目录
