## 1. 调研背景

### 1.1 业务背景

我们计划建设一个面向公司内部的数据中台数据建模工具服务。前期服务对象已经收敛为数据架构师和数仓开发工程师，第一版方向为：

```text
主题域 / 业务对象 / 逻辑模型
-> 物理模型映射
-> 规范检查
-> 模型评审与版本发布
-> 输出模型文档、DDL 草案和开发交接资产
```

本次调研的目的不是直接复制竞品，而是判断市场上成熟产品如何组织对象、流程和能力边界，进一步确认我们的 MVP 应该先做什么、不做什么。

### 1.2 调研问题

1. 国内主流厂商如何定义“数据建模”能力？
2. 建模工具的核心对象是主题域、业务对象、逻辑表、物理表、指标，还是 BI 数据集？
3. 数据架构师和数仓开发之间的协作流程如何被产品化？
4. 哪些能力应进入我们的 MVP，哪些应后置？
5. 我们如何避免做成“大而全数据中台”，而是形成可落地的内部工具闭环？

### 1.3 范围与非目标

| 类型 | 内容 |
|---|---|
| 本次包含 | 数据建模、维度建模、关系建模、BI 语义模型、指标/维度、规范检查、评审发布、模型资产管理 |
| 本次不包含 | 完整数据开发调度、ETL 编排、全链路血缘引擎、数据质量平台、权限中台、商业定价对比 |

## 2. 市场观察

### 2.1 一句话结论

国内数据建模产品正在从“建表开发辅助工具”演进为“模型资产治理 + 指标语义层 + 数据消费闭环”的组合能力。对我们来说，最好的第一步不是追求完整数据中台，而是先把内部数据架构师的模型标准和数仓开发的落地表设计打通。

说明：本次不采用未购买、不可复核的第三方市场规模数字，先以政策驱动、公开产品能力和目标用户痛点作为机会探索依据。

### 2.2 主要驱动

| 驱动 | 说明 | 对我们的影响 |
|---|---|---|
| 数据资产化 | 数据资源入表、数据资产管理和数据要素流通提高了企业对数据口径、责任人、血缘、版本和治理证据的要求。 | 模型不能只是图或表结构，需要成为可追溯、可评审、可发布的资产。 |
| 数据治理前置 | 命名、类型、分层、敏感等级、指标口径等规则如果只在上线后治理，成本高且返工多。 | 规范检查应在模型设计阶段触发，而不是等数据表已经落地后再补治理。 |
| 指标一致性 | 多团队重复定义指标、维度、过滤条件，导致同名不同义或同义不同名。 | 第一版可轻量支持指标/维度描述，但不必马上做完整指标平台。 |
| 研发治理一体化 | 阿里 DataWorks 和字节 / 火山 DataLeap 都强调数据集成、数据开发、任务发布 / 运维、数据地图、数据质量、数据安全、指标 / 服务和智能辅助，说明建模资产会和研发治理、资产消费一起演进。 | 我们 P0 仍聚焦模型协作，但需要预留与数据开发、发布、质量、地图、指标平台和数据服务的连接点。 |
| AI-ready data | AI 分析、智能问数和智能开发依赖稳定的语义、元数据和指标口径。 | AI 能力不宜 P0 强上，但 P0 的结构化模型资产应为后续 AI 留接口。 |

### 2.3 赛道分化

| 路线 | 代表产品 | 典型能力 | 对我们的启示 |
|---|---|---|---|
| 数仓规范建模路线 | 阿里 Dataphin / DataWorks、网易 EasyDesign、华为 DataArts Studio | 主题域、分层、维度/事实、指标、工单/审核、规范检测、质量规则、建设评估 | 最贴近我们的 P0，应优先学习其对象体系和流程闭环。 |
| 关系/ER 建模路线 | 网易关系建模、华为 DataArts Studio、Datablau Data Modeler 相关能力 | 逻辑模型、物理模型、ER 关系、业务对象、模型管理、逻辑转物理 | 可学习其逻辑/物理模型表达和可视化设计，但不要只做画图工具。 |
| 研发治理 / 资产治理路线 | 阿里 DataWorks、火山引擎 DataLeap | 数据集成、数据开发、任务发布 / 运维、数据地图、数据质量、数据服务、指标平台、逻辑表、Git / 版本、发布包、智能开发助手 | 对我们的 P1/P2 扩展很有价值；P0 可借鉴发布、版本、质量、血缘、指标模型和数据服务的衔接方式。 |
| BI 语义模型路线 | 火山引擎 DataWind / Data Fabric | 数据集、字段加工、维度指标、关联模型、权限、分析主题、衍生指标 | 对后续“模型消费层”有参考价值，P0 不宜把主战场放在 BI。 |
| 一体化数据中台路线 | 阿里、网易、华为、字节等平台型产品 | 从建模、开发、治理到分析消费的套件组合 | 对内部工具而言，应该选择窄闭环，不应一开始复制全套平台。 |

## 3. 关键结论

### 3.1 竞品分化

| 竞品 | 核心判断 | 对我们的启示 |
|---|---|---|
| 阿里云 Dataphin / DataWorks | Dataphin 以 Kimball 维度建模和 OneData 思路为基础，强调概念模型、业务对象/业务活动、维度/事实逻辑表、原子/派生指标和汇总逻辑表；DataWorks 则是一站式智能大数据开发治理平台，覆盖数据集成、数据开发、数据治理、数据服务、数据地图、安全和智能辅助，其数据建模支持数仓规划、数据标准、维度建模、数据指标和逆向建模。 | Dataphin 更适合作为 P0 建模对象体系参考；DataWorks 更适合作为 P1/P2 研发治理、资产治理、数据服务和智能化扩展参考。 |
| 网易数帆 EasyDesign / 关系建模 | EasyDesign 强调组织数据、刻画数据、规范约束、效果衡量；关系建模强调逻辑模型、物理模型、ER 关系和业务对象。 | 与我们的“架构师定义模型、开发落地表模型”非常贴近，可作为第一版主要参照。 |
| 华为云 DataArts Studio | 数据架构模块以关系建模、维度建模支撑规范化、可视化、标准化数据模型开发，覆盖主题设计、逻辑模型、关系建模、维度建模、指标设计、审核发布、导入导出和质量规则关联。 | 与我们的 P0 高度贴近，尤其值得学习“逻辑模型转物理模型”“主题发布审核”“发布历史/版本对比”“预览 SQL”和“字段关联质量规则”。 |
| 火山引擎 DataLeap / DataWind / Data Fabric | DataLeap 所在 6260 文档库定位为大数据研发治理套件，覆盖数据集成、数据开发、任务发布、数据治理、数据地图、数据质量、指标平台、逻辑表集市、Git 版本和智能开发助手；DataWind / Data Fabric 更偏分析消费和语义组织。 | DataLeap 可作为研发治理和指标/资产治理参考，DataWind / Data Fabric 可作为 P2 消费层参考；P0 仍先完成数仓模型治理闭环。 |

### 3.2 产品机会

1. **架构到开发的模型交接闭环**：竞品普遍覆盖建模对象，但内部场景更需要把逻辑模型、物理表、评审意见、版本和交付产物串成一条可执行流程。
2. **轻量但强约束的规范检查**：将命名、分层、字段类型、注释、敏感等级、映射完整性前置到模型设计阶段，减少后期治理返工。
3. **版本化模型资产**：每次发布形成主题域、逻辑模型、物理模型、指标/维度说明、DDL 草案和变更 diff 的快照。
4. **不要先做大而全**：调度、ETL、完整血缘、数据质量、权限中台和 BI 语义层都重要，但不是第一版的最小闭环。

## 4. 逐家深剖

### 4.1 阿里云 Dataphin / DataWorks

#### 定位

阿里云需要拆成两条线看：

- **Dataphin**：规范建模明确基于 Ralph Kimball 的维度建模理论。官方文档描述其可根据业务实际在 Dataphin 中设计并创建概念模型，并通过概念模型中的业务实体，即业务对象或业务活动，创建维度表、事实表、原子指标、业务限定、指标和汇总逻辑表。
- **DataWorks**：官方定位是一站式智能大数据开发治理平台，覆盖数据集成、数据开发、数据治理、数据分析 / BI、数据服务、数据地图、数据质量、数据安全、运维管理和 Copilot 等能力。其“智能数据建模”功能支持数仓规划、数据标准、维度建模、数据指标和逆向建模，并可将维度表、明细表、汇总表、应用表物化到不同引擎。

对我们的 P0 来说，Dataphin / DataWorks 的建模对象体系很有价值；对 P1/P2 来说，DataWorks 的研发治理、数据地图、数据质量、数据服务和智能辅助更值得参考。

#### 功能设计

| 模块 | 功能点 | 说明 |
|---|---|---|
| 概念模型 | 业务对象、业务活动 | 作为维度逻辑表和事实逻辑表的上游语义对象。 |
| 维度逻辑表 | 普通维度、层级维度、枚举维度、虚拟维度 | 用于描述业务对象的属性信息。 |
| 事实逻辑表 | 事实逻辑表 | 用于描述业务活动信息。 |
| 指标体系 | 原子指标、衍生原子指标、派生指标、业务限定 | 把计算逻辑、统计周期、业务范围和统计粒度结构化。 |
| 汇总逻辑表 | 派生指标归属表 | 同一统计时效和统计粒度下的派生指标归属到汇总逻辑表。 |
| 研发入口 | 研发 / 数据开发 | 建模能力嵌入数据开发主流程。 |
| DataWorks 数仓规划 | 数据分层、业务分类、数据域、业务过程、数据集市、主题域 | 用规划对象组织数仓建设。 |
| DataWorks 数据标准 | 数据标准、命名词典等 | 支持标准化建模和治理前置。 |
| DataWorks 智能数据建模 | 维度建模、数据指标、逆向建模 | 支持维度表、明细表、汇总表、应用表等模型对象。 |
| DataWorks 研发治理 | 数据集成、数据开发、运维、数据质量、数据地图、数据服务、数据安全、Copilot | 支撑模型资产向开发、治理和消费闭环延伸。 |

#### 使用 / 落地流程

1. 创建或绑定数据板块和项目。
2. 在研发 / 数据开发中进入规范建模。
3. 设计概念模型、业务对象或业务活动。
4. 创建维度逻辑表、事实逻辑表。
5. 定义原子指标、业务限定、派生指标。
6. 形成汇总逻辑表并进入后续开发与发布。
7. 在 DataWorks 侧，可围绕数仓规划、数据标准、维度建模、数据指标、逆向建模、数据开发、数据质量和数据地图形成更完整的研发治理链路。

#### 优势

- 对维度建模对象拆分细，体系成熟。
- 指标定义和研发结合紧，强调“设计即开发”。
- 适合大中型组织统一数据口径和数仓建设方法。
- DataWorks 将数据建模放进数据集成、开发、治理、服务、地图、质量、安全和智能辅助的一体化平台中，适合参考后续扩展边界。
- DataWorks 支持逆向建模和多类表物化，对我们后续“导入已有表结构”和“导出 DDL 草案 / 交接资产”有启发。

#### 不足 / 待验证

- 对小团队或内部轻量工具来说，概念和流程可能偏重。
- 公开文档更强调平台内建模能力，未充分展示跨团队模型评审、轻量交接、DDL 草案边界等内部协作细节。
- DataWorks 能力面很宽，若直接照搬数据集成、开发、运维、质量、地图、服务和安全，会让第一版严重超范围。

### 4.2 网易数帆 EasyDesign / 关系建模

#### 定位

网易 EasyDesign 的维度建模定位为数仓规范化建模工具，提供组织数据、刻画数据、规范约束、效果衡量四类能力。网易关系建模用户手册中还明确出现逻辑模型、物理模型、ER 关系和业务对象等概念，贴近传统数据架构设计与数仓落地之间的连接层。

#### 功能设计

| 模块 | 功能点 | 说明 |
|---|---|---|
| 组织数据 | 主题域、多级主题、表分层 | 用业务视角组织数仓表。 |
| 刻画数据 | 主键、维度、度量、指标、基础字典 | 描述数仓表核心语义。 |
| 规范约束 | 工单设计/修改表、表命名约束、表和字段核心标签 | 将规范嵌入建表和改表流程。 |
| 效果衡量 | 数仓建设水平评估指标体系 | 用指标评估数仓建设成熟度。 |
| 模型设计 | 创建表、导入表、修改表 | 支持数仓表的落地设计。 |
| 关系建模 | 逻辑模型、物理模型、ER 关系、业务对象管理 | 支持架构视角的模型设计和对象管理。 |

#### 使用 / 落地流程

1. 规划主题域、多级主题和表分层。
2. 通过维度、度量、指标和字典刻画表语义。
3. 通过工单新建、导入或修改表。
4. 触发表命名、字段标签等规范约束。
5. 通过规范检测和建设评估反馈数仓建设质量。

#### 优势

- 与我们的用户角色最贴近：数据架构师负责主题域和模型规范，数仓开发通过工单落表。
- 同时覆盖维度建模和关系建模，具备逻辑模型到物理模型的表达基础。
- 规范检测和建设评估值得借鉴。

#### 不足 / 待验证

- 公开资料中对版本发布、变更 diff、DDL 草案、人审边界的说明不够充分。
- 若照搬完整数仓建设平台，第一版容易变重。

### 4.3 华为云 DataArts Studio

#### 定位

DataArts Studio 是华为云的数据治理中心，其“数据架构”模块更贴近本次调研的建模工具方向。官方文档将数据架构定位为基于关系建模、维度建模理论，实现规范化、可视化、标准化的数据模型开发，并将建模成果用于指导开发人员落地数据治理方法论。

DataArts Studio 的数据架构不是单纯 BI 数据集建模，而是覆盖主题设计、数据标准、码表、逻辑建模、关系建模、维度建模、指标设计、数据集市、审核发布、导入导出和质量规则关联的一组治理流程能力。

#### 功能设计

| 模块 | 功能点 | 说明 |
|---|---|---|
| 数据调研 | 流程设计、主题设计、逻辑建模 | 支持主题域创建和发布，逻辑模型可进一步转换为物理模型。 |
| 标准设计 | 码表、数据标准、命名词典 | 码表可用于业务数据标准化、质量监控值域范围和枚举维度。 |
| 模型设计 | 数仓规划、关系建模、维度建模、数据集市 | 支持 ER/关系建模、物理模型、维度、事实表、汇总表等对象。 |
| 指标设计 | 业务指标、技术指标 | 技术指标包含原子指标、衍生指标、复合指标和时间限定等能力。 |
| 审核发布 | 审核人、发布历史、审核历史 | 主题、指标、维度表等对象存在发布审核和历史记录。 |
| 导入导出 | 流程、主题、码表、数据标准、物理表、逻辑实体、维度/事实表、指标、汇总表 | 支持架构资产迁移和批量维护。 |
| 质量关联 | 维度表 / 汇总表字段关联质量规则 | 建模阶段即可关联质量规则，发布后可同步到数据质量模块。 |

#### 使用 / 落地流程

1. 在工作空间进入“数据架构”模块，先配置审核人和流程。
2. 通过主题设计创建主题域或多层主题，并提交审核发布。
3. 在逻辑建模中创建逻辑模型和逻辑实体。
4. 将逻辑模型转换为关系建模中的物理模型，转换时进行前缀等规则校验。
5. 通过关系建模、维度建模和数据集市设计物理表、维度、事实表和汇总表。
6. 定义业务指标和技术指标，提交审核并发布。
7. 查看发布历史、版本对比、预览 SQL，必要时导入导出模型资产或关联质量规则。

#### 优势

- 与我们的目标用户高度匹配，覆盖数据架构师和数仓开发共同关心的主题、标准、逻辑模型、物理模型、维度/事实和指标。
- 明确支持逻辑模型转换为物理模型，这是我们 MVP “架构到开发交接”的强参考。
- 审核发布、发布历史、版本对比和预览 SQL 能力值得直接借鉴。
- 字段级质量规则关联说明其把治理前置到模型设计阶段，符合我们的产品机会判断。
- 导入导出范围覆盖主题、码表、数据标准、物理表、逻辑实体、维度/事实表、指标和汇总表，适合后续存量资产迁移。

#### 不足 / 待验证

- 作为平台型产品，能力范围较大，直接照搬会让我们的第一版过重。
- 公开资料中对评审评论、模型变更 diff、开发交接说明等轻量协作体验展示不多。
- 部分同步/发布能力可能会触达数据库实际对象；我们的 MVP 仍应只生成 DDL 草案，不自动执行迁移。

### 4.4 火山引擎 DataLeap / DataWind / Data Fabric

#### 定位

火山引擎需要拆成两条线看：

- **DataLeap / 大数据研发治理套件（6260 文档库）**：更偏数据研发治理，覆盖数据集成、数据开发、任务发布、数据治理、数据地图、数据质量、指标平台、逻辑表集市、元数据、物理表管理、数仓建表规范、Git 版本、Pipeline / Notebook / SQL 开发、发布包和智能开发助手等能力。
- **DataWind / Data Fabric（4726 文档库）**：更偏分析消费和语义组织，覆盖数据集、Fabric 模型、分析主题、关联数据集、衍生指标、数据门户和 AI 能力。

对我们的建模工具来说，DataLeap 比 DataWind 更应该进入竞品主分析，因为它更接近“数仓开发 + 数据治理 + 指标资产 + 发布运维”的平台化工作流。

#### 功能设计

| 模块 | 功能点 | 说明 |
|---|---|---|
| 数据集成 | 数据源、集成资源组、实时/离线集成、解决方案表映射 | 支撑模型落地后的数据接入和同步。 |
| 数据开发 | SQL 开发、Notebook、Pipeline、任务模板、调度属性、历史版本、Git 版本管理 | 面向数仓开发的研发工作台。 |
| 任务发布 | 待发布、发布包管理、发布复查策略 | 把开发产物纳入发布治理。 |
| 数据地图 / 资产 | 元数据、物理表管理、我的资产、资产目录、资产专辑、标签管理、表血缘 | 支撑模型资产的发现、消费和影响分析。 |
| 数据质量 | 数据质量规则、双数据源校验、规则运行结果、治理任务 SLA | 将质量治理和研发/资产闭环关联。 |
| 指标平台 | 指标分类、原子指标、衍生指标、复合指标、指标/维度、模型分类、创建模型、配置并发布模型 | 支撑指标语义和模型发布管理。 |
| 逻辑表集市 | 逻辑表管理 | 提供逻辑表层面的组织和管理入口。 |
| 智能化能力 | DataLeap 智能开发助手、数据地图助手、SQL Insight、代码版本助手 | 面向研发、地图和排障的智能辅助。 |
| DataWind / Data Fabric | 数据集、Fabric 模型、分析主题、关联数据集、衍生指标、数据门户 | 偏分析消费和语义层，可作为后续扩展参考。 |

#### 使用 / 落地流程

1. 通过 DataLeap 接入数据源并管理物理表、元数据和数据地图资产。
2. 数仓开发在 SQL / Notebook / Pipeline 等环境中开发任务，并通过调度、历史版本、Git 和发布包治理研发过程。
3. 通过数据质量、数据地图、标签、表血缘和资产目录沉淀治理与消费信息。
4. 在指标平台中管理原子指标、衍生指标、复合指标、指标/维度和指标模型，并配置发布模型。
5. 后续可通过 DataWind / Data Fabric 将数据集、分析主题、衍生指标和数据门户连接到分析消费层。

#### 优势

- DataLeap 把数据开发、任务发布、资产地图、数据质量、指标平台放在同一研发治理套件中，体现了“模型不是孤立设计资产，而是研发治理资产”的趋势。
- 指标平台中出现原子指标、衍生指标、复合指标、指标/维度、模型分类、创建模型、配置并发布模型等对象，适合我们 P1 指标增强参考。
- 数据地图、物理表管理、表血缘、资产目录、标签管理对我们后续模型影响分析和资产消费有直接参考价值。
- Git 版本、历史版本、发布包管理、发布复查策略说明其重视研发过程和发布治理，值得借鉴到我们的模型版本发布。
- 智能开发助手、数据地图助手、SQL Insight 说明后续 AI 能力应基于结构化资产和研发上下文，而不是脱离流程独立存在。

#### 不足 / 待验证

- DataLeap 是平台级研发治理套件，范围远超建模工具；P0 不能直接复制数据集成、调度、发布、质量、资产地图全套能力。
- 公开目录中能看到指标模型、逻辑表、物理表、数据地图和质量治理，但“数据架构师先定义企业逻辑模型、数仓开发再映射物理模型”的显式流程不如华为 DataArts Studio 清晰。
- DataWind / Data Fabric 的分析消费能力对后续有价值，但不是我们第一版的主战场。

## 5. 机会点

| 机会点 | 用户痛点 | 竞品现状 / 缺口 | 实现难度 | 优先级 |
|---|---|---|---|---|
| 逻辑模型到物理模型映射 | 架构师的逻辑模型常停留在文档，开发落表时口径和字段容易漂移。 | 竞品覆盖逻辑/物理或维度/事实，但公开资料中轻量交接闭环不突出。 | 中 | P0 |
| 规范检查前置 | 命名、字段类型、注释、分层、敏感等级常在上线后补治理。 | 阿里 Dataphin / DataWorks、网易、华为均有规范/标准/质量前置能力；我们应做轻量可配置规则。 | 中 | P0 |
| 模型评审与版本发布 | 模型变更缺少 diff、评论、结论和发布快照。 | 竞品有工单/审批迹象，但内部协作可做得更贴近研发流程。 | 中 | P0 |
| 发布产物生成 | 开发交接靠人工整理文档、DDL、字段清单。 | 阿里强调设计即开发；我们可先生成文档和 DDL 草案，不自动执行。 | 中 | P0 |
| 指标/维度轻量管理 | 口径不一致，指标散落在代码、报表和文档里。 | 阿里 Dataphin / DataWorks、DataLeap 指标平台能力较强；我们 P0 只做模型内描述，P1 再扩展。 | 中 | P1 |
| 研发治理衔接 | 模型发布后如果不能进入开发、发布、质量、资产地图和血缘体系，会重新回到文档孤岛。 | DataWorks 和 DataLeap 在研发治理链路上较完整；我们 P0 先预留接口，P1/P2 再对接。 | 高 | P1 |
| BI 语义消费 | 业务分析需要统一数据集和指标语义。 | DataWind / Data Fabric 相关能力较成熟；华为 BI 分析产品线如需纳入，应另行调研。 | 高 | P2 |
| AI 建模助手 | 架构师和开发希望基于已有表/字段推荐模型和规范问题。 | 市场趋势明显，但依赖高质量元数据。 | 高 | P2 |

## 6. MVP 建议

### 6.1 MVP 目标

第一版帮助内部数据架构师和数仓开发工程师，在一个主题域内完成从逻辑模型到物理模型的协作闭环，并在发布时形成可审计、可交接、可复用的模型资产。

### 6.2 MVP 功能范围

| 模块 | MVP 能力 | 验收标准 |
|---|---|---|
| 主题域管理 | 主题域、负责人、边界、包含业务对象 | 可创建主题域并维护业务边界和责任人。 |
| 业务对象 / 逻辑模型 | 业务对象、逻辑实体、逻辑字段、字段属性、实体关系 | 架构师可完成一个业务对象的逻辑模型定义。 |
| 物理模型映射 | 表、字段、分区、主键、逻辑字段映射、DDL 草案 | 数仓开发可基于逻辑模型设计物理表，并查看映射覆盖率。 |
| 规范检查 | 命名、类型、注释、必填元数据、敏感等级、映射完整性 | 提交评审前能阻断 P0 级规则问题。 |
| 模型评审 | 提交、评论、通过、驳回、变更 diff | 架构师和开发可围绕一次模型变更完成评审。 |
| 版本发布 | 发布快照、版本号、发布说明、文档导出、DDL 草案导出 | 发布后能生成模型文档和开发交接资产。 |

### 6.3 暂不进入 MVP

1. 数据开发调度和 ETL 编排。
2. 自动执行数据库迁移脚本。
3. 完整血缘分析。
4. 企业级权限中台。
5. 完整指标平台。
6. BI 数据集/语义层消费。
7. AI 自动建模。

### 6.4 推荐路线

第一阶段：模型协作闭环

- 主题域、业务对象、逻辑模型、物理模型映射、规范检查、评审、发布产物。

第二阶段：指标与元数据增强

- 指标/维度一等对象、字段标准库、字典、存量表导入、模型 diff、发布影响分析。

第三阶段：消费与智能化

- BI 数据集映射、语义层、基础血缘、AI 建模建议、自然语言问数所需语义资产。

## 7. Grilling 问题清单

进入 PRD 前建议继续澄清：

1. 第一版主题域样例选哪个？客户域、产品域、交易域还是资产域？
2. 逻辑模型和物理模型的必填字段有哪些？
3. 字段标准、字典、敏感等级是否已有公司内部规范？
4. 物理模型目标数据库优先支持哪些？Hive、Doris、ClickHouse、MySQL、StarRocks、PostgreSQL？
5. DDL 草案只导出文本，还是需要对接工单系统？
6. 评审流是单人架构师审批，还是架构师 + 数仓负责人双审批？
7. 发布后模型能否修改？修改是否必须新建版本？
8. 规范检查中哪些问题必须阻断，哪些只警告？
9. 是否需要导入已有表结构作为物理模型起点？
10. MVP 成功指标是建模效率、规范问题减少、交接时间缩短，还是模型资产覆盖率？

## 8. 建议下一步

1. 基于本文档进入 `grill-with-docs`，把用户、场景、MVP、非目标范围和验收标准压实。
2. 选择一个真实主题域作为 PRD 样例，避免抽象建模工具脱离业务。
3. 产出 PRD 初稿后，进入 `product-design-prototype` 设计核心页面和状态矩阵。
4. 有 UI 后先做原型评审，再进入 OpenAPI Draft。
