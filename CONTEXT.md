# 领域上下文

本文档是项目的领域词汇表，只记录业务语言和协作概念，不记录实现细节、计划草稿或架构说明。

## 术语表

| 术语 | 含义 | 避免 / 备注 |
|---|---|---|
| OpenSpec | 基于 OpenAPI 3.1 的 API 契约层。 | 讨论前后端 API 契约时使用。 |
| Agent | 执行特定工作流步骤的 AI 协作者。 | 不要与人工审查者混用。 |
| PRD | 产品需求文档，用于记录用户问题、解决方案、用户故事、关键决策和测试 seam。 | 除非作为既有参考，否则不要写入具体实现文件路径。 |
| 垂直切片（Vertical Slice） | 贯穿所有受影响层、可独立验证的窄功能路径。 | 优先使用垂直切片，避免只按层拆分的横向任务。 |
| ADR | 架构决策记录，用于沉淀难以回滚、非显而易见且存在真实取舍的技术决策。 | 常规实现选择不要写 ADR。 |
| 企业数据模型协作工作台 | 面向内部数据架构师和数仓开发工程师的模型协作工具，用于沉淀、评审、发布和交接数据模型资产。 | 不等同于完整数据中台、数据开发调度平台或 BI 分析平台。 |
| 模型项目 | 工作台中承载一个业务数据模型建设目标的顶层协作单元，可包含一个或多个主题域、子域、业务对象和模型资产。 | 不等同于主题域；绩效风控指标模型只是多个模型项目之一。 |
| 数据架构师 | 负责主题域边界、业务对象、逻辑模型、建模规范和模型评审结论的角色。 | 不默认承担具体 ETL 任务开发。 |
| 数仓开发工程师 | 基于逻辑模型设计物理表、字段映射和 DDL 草案，并根据评审意见修正落地方案的角色。 | 不等同于业务分析师或 BI 报表开发者。 |
| 主题域 | 从业务视角划分的一组相关业务对象和模型资产边界。 | 不是 BI 分析主题，也不是数据库 schema 的同义词。 |
| 业务对象 | 业务上可被稳定识别和描述的核心对象，是逻辑模型组织的基本语义单元。 | 不直接等同于物理表；一个业务对象可以映射到多个物理表。 |
| 逻辑模型 | 描述业务对象、逻辑实体、字段语义和关系的模型。 | 不记录具体数据库存储属性。 |
| 物理模型 | 描述物理表、字段、分区、主键和存储相关属性的模型。 | 不代表已在数据库中执行变更。 |
| 模型映射 | 逻辑模型中的实体或字段与物理模型中的表或字段之间的对应关系。 | 映射不完整时需要在评审中显式说明原因。 |
| 规范检查 | 在模型提交评审或发布前，对命名、分层、字段类型、注释、敏感标记、待分级状态和映射完整性等规则进行检查。 | 不等同于运行时数据质量校验；正式敏感分级体系后置。 |
| 模型评审 | 围绕一次模型变更进行评论、修正、通过或驳回的协作过程。 | 不等同于完整研发发布审批。 |
| 发布快照 | 模型发布时形成的不可变版本记录，包含当时的逻辑模型、物理模型、映射、检查结果和发布说明。 | 快照用于追溯，不表示自动执行数据库迁移。 |
| DDL 草案 | 由物理模型生成的建表或改表 SQL 草稿，用于开发交接和人工确认。 | 不自动执行；涉及数据库迁移时必须人工审查。 |

只有在计划、分诊、调试或架构讨论中明确沉淀出稳定语言时，才新增术语。
